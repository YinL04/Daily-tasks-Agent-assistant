## 1. 文档信息

- 产品名称：Personal Agent Planner
- 文档版本：v0.1
- 文档日期：2026-05-22
- 产品阶段：本地可运行 MVP
- 目标读者：产品、设计、研发、测试、个人效率工具使用者

## 2. 产品概述

Personal Agent Planner 是一个本地运行的个人事务规划 Agent。用户用自然语言输入目标、待办、旅行准备、学习计划、链接整理或文件生成需求后，系统通过上下文构建、长期记忆检索、ReAct 工具调用循环和一组可扩展 Skills，将输入转化为结构化任务、日历事件、参考链接、执行建议、运行日志和可下载计划文件。

产品不是单纯的聊天助手，而是一个面向个人事务落地的工作台：它既给出自然语言总结，也保留可编辑、可追踪、可复用的结构化结果。

## 3. 背景与问题

个人规划类需求通常有以下痛点：

- 用户的原始想法是模糊的自然语言，难以直接变成可执行待办。
- 任务之间存在优先级、依赖关系和时间安排，但手工拆解成本较高。
- 旅行、学习、项目推进等场景需要同时整理任务、日程、链接和文档。
- 常规聊天工具输出后难以追踪执行过程，也不方便沉淀为日历或文件。
- AI 输出存在不稳定性，需要通过工具白名单、执行日志和本地持久化增强可控性。

Personal Agent Planner 通过 Agent Runtime 将“理解意图”和“可控执行”分离：LLM 负责自然语言理解、拆解和建议，本地代码负责校验、排序、存储、文件生成和日志记录。

## 4. 产品目标

### 4.1 当前阶段目标

- 支持用户输入自然语言目标，并生成结构化任务清单。
- 根据任务优先级、截止日期和依赖关系生成可编辑日历事件。
- 整理用户提供或系统建议的参考链接。
- 生成 Markdown、JSON、CSV、ICS 等计划文件。
- 展示一次 Agent Run 的 ReAct 执行日志，提升可解释性。
- 保存运行历史，方便用户回看过往规划结果。
- 支持本地长期记忆，用于增强后续规划上下文。
- 在 LLM 未连接时清晰展示状态，避免用户误以为获得完整 AI 分析。

### 4.2 非目标

- 暂不做多人协作、团队权限、云同步或账号体系。
- 暂不做真实外部日历双向同步，如 Google Calendar、Outlook Calendar。
- 暂不承诺实时价格、票务、政策、交通班次等不稳定信息的准确性。
- 暂不把记忆管理作为前端主功能入口，记忆主要作为 Agent 上下文增强层。
- 暂不做复杂项目管理能力，如甘特图、资源负载、跨项目容量规划。

## 5. 目标用户

### 5.1 核心用户

- 独立开发者、学生、知识工作者：需要把学习、项目、写作、旅行等目标快速拆成可执行计划。
- AI Agent 学习者和开发者：希望观察 ReAct、Skills、Memory、Harness 等 Agent 工程模式如何落地。
- 重度本地工具用户：希望数据留存在本地，并能导出文件继续使用。

### 5.2 用户画像

1. 学习规划用户
   - 输入：“帮我规划下周 TypeScript 学习，每天 2 小时，周三前完成一个 React 小项目。”
   - 期望：得到任务拆解、学习节奏、日历安排、参考入口和计划文件。

2. 旅行准备用户
   - 输入：“我要准备一次大阪旅行，帮我安排订机票、订酒店、预算、查景点和准备行李。”
   - 期望：得到前置依赖明确的准备清单、日历提醒、风险提示和待调研链接。

3. Agent 工程学习者
   - 输入：任意复杂目标。
   - 期望：看到每一步 Thought / Action / Observation，理解系统为何调用某个 Skill。

## 6. 使用场景

### 6.1 任务规划

用户输入一个自然语言目标，系统自动拆解成 3-10 个可执行任务，每个任务包含标题、描述、优先级、预计耗时、依赖、标签、状态和可选截止日期。

### 6.2 日历安排

系统将任务转换为本地日历事件，用户可以在前端新增、修改和删除事件。历史记录中的日历事件以只读快照形式展示。

### 6.3 链接整理

系统提取用户输入中的 URL，并根据任务场景生成参考入口。对于旅行等实时性较强场景，链接和建议应明确表达“候选”“待核验”“需进一步查询”，避免编造实时事实。

### 6.4 文件生成

当用户要求生成文件，或 Agent 在 ReAct 循环中选择文件生成工具时，系统输出 Markdown 计划、JSON 结果、CSV 任务清单和 ICS 日历文件。

### 6.5 历史回看

用户进入历史记录页面，可以查看每次运行的输入、最终总结、任务数量、日历事件数量、文件数量，并打开单次运行详情。

## 7. 核心用户流程

### 7.1 新建规划流程

1. 用户进入工作台。
2. 系统自动检测 LLM 连接状态。
3. 用户输入自然语言目标。
4. 用户点击运行。
5. 后端构建 AgentContext，包括 runId、原始输入、当前时间和相关记忆。
6. AgentExecutor 进入 ReAct 循环。
7. LLM 从 allowedActions 中选择下一步 Action。
8. AgentHarness 执行对应 Skill，并记录 Observation。
9. 系统聚合任务、日历、链接、建议、文件和日志。
10. 前端展示执行结果，日历事件保存到本地。

### 7.2 编辑日历流程

1. 用户在工作台查看日历事件。
2. 用户新增、修改或删除事件。
3. 前端调用日历 API。
4. 后端更新本地 calendar store。
5. 前端刷新日历视图。

### 7.3 查看历史流程

1. 用户进入历史记录页面。
2. 系统加载运行列表。
3. 用户点击某条历史运行。
4. 系统展示该次运行的总结、任务、链接、日历快照和生成文件。

## 8. 功能需求

### 8.1 工作台输入

- 用户可以输入任意自然语言目标。
- 输入不能为空；为空时应阻止运行或返回清晰错误。
- 输入可以包含目标、待办、时间约束、地点、链接和文件生成要求。
- 运行过程中按钮应进入 loading 状态，避免重复提交。

验收标准：

- 输入有效文本后，可以触发 `POST /api/agent/run`。
- 运行失败时，前端展示可读错误信息。
- 运行成功后，展示结构化结果。

### 8.2 LLM 连接状态

- 工作台需要展示 LLM 是否连接、模型名称、Provider 和延迟。
- 用户可以手动重新检测连接。
- 当 LLM 不可用时，界面应明确说明当前不是完整 AI 分析。

验收标准：

- 页面加载后调用 `GET /api/agent/llm-status`。
- 连接成功时显示已连接、模型和延迟。
- 连接失败时显示错误原因和环境变量配置提示。

### 8.3 Agent ReAct 执行

- 系统采用 Thought -> Action -> Observation 的 ReAct 模式。
- LLM 只能选择当前 allowedActions 中的工具。
- 每一步执行都应生成结构化日志。
- Harness 应限制最大执行步数，防止无限循环。
- 工具执行失败时应记录错误 Observation。

验收标准：

- 每次运行结果包含 `agentPattern: "react"`。
- 每个 step 包含 id、name、status、inputSummary、startedAt，并尽量包含 thought、action、observation、usedLLM。
- 前端能展示 ReAct steps。

### 8.4 任务拆解

- 系统将用户目标拆成结构化任务。
- 任务字段包括 id、title、description、priority、estimatedMinutes、dependencies、tags、status。
- 只有当用户提供明确日期或可合理推断时，才生成 dueDate。
- 任务优先级限定为 high、medium、low。

验收标准：

- 返回 `PlannedTask[]`。
- 每个任务有标题、描述、优先级、预计耗时和状态。
- 无效优先级、过大耗时、非法日期需要被清洗或丢弃。

### 8.5 优先级排序

- 系统根据优先级、截止日期和依赖关系对任务排序。
- 高优先级、临近截止、被其他任务依赖的任务应靠前。

验收标准：

- 排序后的任务列表稳定展示。
- 依赖关系不应导致明显反直觉顺序，例如后置交付早于前置准备。

### 8.6 日历规划

- 系统可以将任务转换为日历事件。
- 日历事件包含 id、taskId、title、start、end、priority、description。
- 用户可以在前端新增、编辑和删除本地日历事件。
- 历史运行详情中的日历事件作为快照展示。

验收标准：

- Agent 运行后可保存日历事件。
- `GET /api/calendar` 能读取当前本地日历。
- `POST /api/calendar`、`PATCH /api/calendar/:id`、`DELETE /api/calendar/:id` 可正常修改事件。

### 8.7 链接整理

- 系统提取用户输入中的链接。
- 系统可以根据任务生成建议参考入口。
- 链接分类包括 user_provided、suggested、research_needed。
- 对实时信息应提醒用户二次核验。

验收标准：

- 返回 `UrlReference[]`。
- 每条链接包含 title、url、reason、category。
- 对无法确定的真实链接，不应伪造具体 URL。

### 8.8 建议与总结

- 系统应生成 2-3 句最终总结。
- 系统应生成最多 6 条执行建议。
- 建议应结合原始输入、任务和日历事件。

验收标准：

- `finalAnswer` 非空。
- `recommendations` 为字符串数组。
- 建议不应包含未经核验的实时事实。

### 8.9 文件生成

- 系统支持生成 Markdown、JSON、CSV、ICS 文件。
- 文件保存到 `generated/plans` 和 `generated/exports`。
- 前端展示文件名称、类型和下载入口。

验收标准：

- 返回 `GeneratedFile[]`。
- 文件类型限定为 markdown、json、csv、ics。
- 下载 URL 可以访问对应文件。

### 8.10 运行历史

- 每次 Agent Run 保存输入、结果、任务、日历事件、链接、文件和创建时间。
- 历史列表展示摘要。
- 历史详情展示完整结果快照。

验收标准：

- `GET /api/runs` 返回历史摘要。
- `GET /api/runs/:runId` 返回单次运行详情。
- 前端历史页可以打开并展示详情。

### 8.11 长期记忆

- 后端支持记忆 CRUD。
- Agent 运行前检索相关记忆并写入 AgentContext。
- Agent 可以从用户输入中推断稳定偏好，但推断记忆需要使用较低置信度。
- 记忆默认不作为前端主界面能力展示。

验收标准：

- `GET /api/memories`、`POST /api/memories`、`PATCH /api/memories/:id`、`DELETE /api/memories/:id` 可用。
- Agent Run 结果包含 `memoriesUsed`。
- 推断记忆需要标记 `source: "agent_inferred"`。

### 8.12 Skills 扩展

- 每个 Skill 应由 Markdown 定义和 TypeScript runtime 组成。
- Markdown 定义描述职责、输入输出和规则。
- Runtime 负责实际执行、校验和异常处理。
- 新增 Skill 后需要注册到 skills index 和 executor runtime。

验收标准：

- `GET /api/skills` 可查看已加载 Skill 定义。
- 新增 Skill 时不影响现有 Agent Run。
- Skill 失败时不会让整个服务进程崩溃。

## 9. 数据结构

### 9.1 Agent Run Result

```ts
interface HarnessRunResult {
  runId: string;
  agentPattern: "react";
  steps: AgentStepLog[];
  finalAnswer: string;
  tasks: PlannedTask[];
  calendarEvents: CalendarEvent[];
  urls: UrlReference[];
  files: GeneratedFile[];
  memoriesUsed: MemoryItem[];
  recommendations: string[];
  llmStatus?: LLMStatus;
}
```

### 9.2 核心实体

- PlannedTask：任务。
- CalendarEvent：日历事件。
- UrlReference：链接引用。
- GeneratedFile：生成文件。
- MemoryItem：长期记忆。
- AgentStepLog：ReAct 执行日志。
- LLMStatus：模型连接状态。

## 10. 页面需求

### 10.1 工作台

页面内容：

- 页面标题和产品说明。
- LLM 连接状态面板。
- 自然语言输入框。
- 本地日历视图。
- 执行结果总结。
- AI 使用报告。
- 任务列表。
- 链接列表。
- 生成文件面板。
- ReAct 执行日志。

关键状态：

- 初始状态：展示连接状态和输入框。
- 运行中：输入提交按钮 loading。
- 成功：展示完整结果。
- 失败：展示错误。
- LLM 未连接：展示降级提示。

### 10.2 历史记录

页面内容：

- 运行列表。
- 每条记录展示输入摘要、最终总结、创建时间、任务数、日历事件数和文件数。
- 点击后展示记录详情。

关键状态：

- 加载失败时展示错误。
- 无历史记录时应展示空状态提示。

## 11. API 需求

- `POST /api/agent/run`：运行一次 Agent。
- `GET /api/agent/llm-status`：检测 LLM 连接。
- `GET /api/files`：列出生成文件。
- `GET /api/files/:filename`：下载生成文件。
- `GET /api/skills`：查看已加载 Skill。
- `GET /api/runs`：查看运行历史摘要。
- `GET /api/runs/:runId`：查看单次运行详情。
- `GET /api/calendar`：读取本地日历事件。
- `POST /api/calendar`：新增本地日历事件。
- `PATCH /api/calendar/:id`：编辑本地日历事件。
- `DELETE /api/calendar/:id`：删除本地日历事件。
- `GET /api/memories`：读取记忆。
- `POST /api/memories`：新增记忆。
- `PATCH /api/memories/:id`：编辑记忆。
- `DELETE /api/memories/:id`：删除记忆。

## 12. 非功能需求

### 12.1 可用性

- 服务在未配置 LLM_API_KEY 时仍可启动。
- 前端需要清晰展示 LLM 不可用状态。
- 错误信息应面向用户可理解，不直接暴露敏感配置。

### 12.2 可解释性

- 每次 Agent Run 需要展示执行日志。
- 日志需要区分 AI 步骤和本地步骤。
- 工具调用结果应形成 Observation，便于回溯。

### 12.3 可扩展性

- 新 Skill 应能以 Markdown 定义 + TypeScript runtime 的模式接入。
- AgentExecutor 中的工具选择、输入、输出写回和可用条件应保持集中管理。

### 12.4 数据本地化

- 运行历史、记忆、日历和生成文件存储在本地。
- 默认不上传用户历史数据到第三方服务，LLM 调用除外。

### 12.5 安全与隐私

- 不应把 API Key、`.env` 内容或敏感信息写入生成文件、日志或记忆。
- Agent 推断记忆时需要谨慎，避免保存一次性、敏感或不稳定信息。
- 生成旅行、票务、政策、价格等建议时，需要提醒用户核验。

### 12.6 性能

- 普通 Agent Run 应在用户可接受时间内完成。
- LLM 网络失败或超时时应返回清晰错误或降级结果。
- 历史记录和本地文件读取不应阻塞主流程过久。

## 13. MVP 验收标准

- 用户可以在工作台输入一段自然语言目标并成功运行。
- 系统返回最终总结、任务、建议、链接、日历事件、执行日志。
- 用户可以新增、编辑、删除日历事件。
- 用户可以在历史页面查看过往运行。
- 系统可以展示 LLM 连接状态。
- 配置可用 LLM 后，任务拆解、建议和总结明显使用模型能力。
- 未配置 LLM 时，服务可启动，前端明确展示未连接状态。
- 文件生成能力可输出计划文档和结构化导出文件。
- ReAct 日志能展示 Thought、Action、Observation。

## 14. 质量与测试要求

- 类型检查：运行 `npm run typecheck`。
- 构建验证：运行 `npm run build`。
- 服务端测试：运行 `npm run test`。
- Agent Harness 测试应覆盖成功、失败、最大步数限制。
- React 质量测试应覆盖关键页面或组件的基础渲染。
- 手工测试至少覆盖学习计划、旅行准备、文件生成、LLM 未连接四类场景。

## 15. 风险与应对

| 风险               | 影响                         | 应对                                              |
| ------------------ | ---------------------------- | ------------------------------------------------- |
| LLM 输出格式不稳定 | 任务、链接、建议解析失败     | 使用 JSON 生成约束、重试、字段清洗和 fallback     |
| ReAct 循环失控     | 运行时间过长或重复调用       | Harness 限制最大步数，allowedActions 白名单       |
| 实时信息幻觉       | 旅行、价格、政策建议误导用户 | Prompt 约束不编造，标记待核验，避免具体不稳定事实 |
| 记忆误存           | 后续规划受到错误偏好影响     | 推断记忆低置信度，避免敏感信息，提供后端 CRUD     |
| 本地文件增长       | 历史和导出文件占用空间       | 后续增加清理、归档和删除能力                      |
| LLM 配置门槛       | 用户无法获得完整体验         | 前端显示配置提示，README 保留环境变量说明         |

## 16. 后续规划

### 16.1 P1

- 增加历史记录删除和生成文件清理能力。
- 增加前端记忆管理入口，但默认隐藏高级选项。
- 提供 Agent Run 的更清晰状态流转，如排队、运行中、完成、失败。
- 增强空状态、错误态和 loading 态。
- 增加更多业务级 fallback，降低 LLM 不可用时的断裂感。

### 16.2 P2

- 支持外部日历导入导出或单向同步。
- 支持模板化规划场景，如学习、旅行、项目、健康习惯。
- 支持用户在运行前选择是否生成文件、是否生成日历、是否使用记忆。
- 支持编辑任务状态，并将状态变更写入历史或当前计划。
- 支持更细粒度的 Skill 配置和调试视图。

### 16.3 P3

- 支持向量记忆检索。
- 支持多模型 Provider 配置。
- 支持长期目标追踪和周期性复盘。
- 支持端到端导出完整计划包。
- 支持更丰富的可视化，如时间线、依赖图和周视图。

## 17. 成功指标

- 首次有效规划完成率：用户输入后能获得可用任务和总结的比例。
- 结构化结果使用率：用户查看或编辑任务、日历、文件的比例。
- 历史回看率：用户打开历史记录详情的次数。
- 文件导出率：用户生成并下载计划文件的比例。
- LLM 配置成功率：配置模型后连接检测成功的比例。
- 运行失败率：Agent Run 返回错误的比例。

## 18. 开放问题

- 是否需要把“记忆管理”正式暴露给普通用户，还是继续作为开发者能力保留？
- 文件生成应由 Agent 自主决定，还是由用户在运行前明确勾选？
- 日历事件是否应该支持冲突检测、工作时间段和用户偏好时间？
- 历史记录是否需要支持搜索、标签和删除？
- 是否需要引入外部搜索能力来处理旅行和资料调研场景？
- 是否需要为不同场景提供独立 Prompt 模板和评测集？
